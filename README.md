# c-stdlib-getenv-config-snapshot-lab

Small deterministic C/POSIX correctness lab for `getenv()`, `setenv()`, `unsetenv()`, `putenv()`, sequential pointer snapshots, bounded copying, explicit configuration precedence, and child environment-vector construction.

## Scope

- `getenv()` returns library-owned storage (ISO C)
- `setenv()` / `unsetenv()` mutate global process state (POSIX)
- `putenv()` retains caller-owned writable storage (POSIX)
- Startup snapshots, bounded copying, explicit precedence
- Bounded integer configuration parsing
- Child environment-vector construction (no execve executed)

**Does NOT:** create threads, race environment variables, dereference invalidated getenv pointers, train models, benchmark, validate ML.

### Thread summary

**pitdicker** connected environment mutation to Rust time libraries: chrono and time-rs relied on `localtime_r`, which reads `TZ`. Rust declared environment mutation safe, POSIX declared it unsafe. CVEs: CVE-2020-26235, RUSTSEC-2020-0071, RUSTSEC-2020-0159. Chrono parsed timezone databases natively with Rust locks; time-rs panicked on multi-threaded access without libc thread-safety guarantees.

**lifthrasiir** clarified: affected were chrono and time-rs 0.1; time-rs 0.2+ rewrote and lacked general timezone support initially. Noted broader timezone-API problems: `/etc/localtime` polling, `stat()` per `localtime()` call, no subscription mechanism for timezone changes. (Disclaimer: original author of Chrono.)

**Cross-language locking:** multiple commenters (including a thread discussing Rust/Go/C FFI) noted a lock inside one language runtime (Rust `std::env::set_var`, Go `os.Setenv`) cannot protect calls made through other languages or native libraries linked into the same process. C code calling `setenv(3)` bypasses Rust's lock.

**the_mitsuhiko**: a truly shared lock would require changing the standard or common interface — "it cannot without changing the standard. There is really nothing anyone can do without a change in the standard."

**cryptonector** pointed to Solaris/Illumos as proof that safer implementation strategies exist. Clarified that `putenv()` in Solaris/Illumos is thread-safe too, and direct `environ` writes are allowed (with getenv locking). Discussed memory cost of retaining old strings. Said a caller-owned copying interface such as a hypothetical `getenv_r` would require a new API rather than silently changing `getenv` ownership.

**yason** argued environment variables are normally startup configuration rather than inter-thread communication. Recommended reading and caching them early. "Environment variables are not meant to be an inter-thread communication channel." Use `execve()` with `envp` for child processes. Parse at startup, use internal variables for runtime state.

**Joker_vD** objected: applications may contain threads started by third-party libraries before user code reaches `main()`. "just don't call setenv() after creating threads" is not a solution when you don't control all code in the process.

**xxs** argued mechanically thread-safe access would not make mutable global configuration semantically coherent. "Even if all the calls become thread safe, the behavior would be inconsistent, still."

**hddqsb** said reading one environment variable while another thread writes a different variable can still be a reasonable use case. "It is perfectly reasonable and consistent for one thread to set an environment variable while other threads are reading *different* environment variables."

**jeroenhd** argued providing runtime mutation functions invites their use even when the interface remains a footgun. Noted Windows `GetEnvironmentVariable` copies to caller-supplied buffer, avoiding the issue. Suggested languages like Go/Rust should implement their own environment system rather than wrapping POSIX's API.

**jstimpfle** defended using existing interfaces within documented restrictions. "It can and should be used in the cases where it makes sense, with the restrictions that are documented. […] if those APIs were removed it would hurt a lot of old software that is running perfectly fine." Noted `getenv()` returning a pointer to memory invalidated by `setenv()` is fundamentally thread-unsafe. Suggested alternative thread-safe API with RW mutex and user-provided buffer, but acknowledged existing software wouldn't use it. "Use setenv() only at startup and after fork() and before exec(), done."

**andrewla** said the `getenv` return-value interface must be addressed before mutation operations can be made robust. "You simply cannot make setenv() thread-safe so long as getenv() has its current interface. You need to make getenv() safe first; ideally with a getenv_r() call that fills in a user-supplied buffer."

The thread does **not** establish that every libc has the same failure mode, that every use of getenv is unsafe, that environment variables should be used for thread communication, that leaking old values is cost-free, that one mutex in one runtime solves cross-language access, or that one replacement API is universally preferable.

## Standards

- ISO C: `getenv` returns pointer to library-owned storage, may be invalidated by subsequent calls
- POSIX: `getenv`, `setenv`, `unsetenv`, `putenv`, `environ`, overwrite behavior, invalid names, returned pointers, thread-safety classifications — see pubs.opengroup.org
- C11 Annex K: optional `getenv_s`
